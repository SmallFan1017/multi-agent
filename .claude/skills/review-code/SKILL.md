# /review-code — 程式碼審查

審查使用者提供的程式碼，確認是否符合需求，並找出潛在問題。

## 使用方式

```
/review-code
```

執行前，請在對話中提供：
1. **程式碼**：直接貼上，或指定檔案路徑
2. **需求描述**：這段程式碼應該做什麼

## 執行流程

> 靈感來自官方 `/simplify`：平行啟動多個審查視角，最後彙整結果。

**Step 1 — 理解需求**
確認使用者描述的需求。若需求不明確，先詢問釐清再進行審查。

**Step 2 — 三向平行審查**（各自獨立評估）

| 審查維度 | 重點 |
|---------|------|
| 需求符合性 | 程式碼是否完整實現了所有描述的功能？ |
| 邏輯與邊界 | off-by-one、null reference、型別錯誤、邊界條件 |
| 潛在風險 | 效能瓶頸、資源洩漏、未處理例外、競態條件 |

**Step 3 — 動態驗證**（若可執行）
- 撰寫最小測試案例驗證核心邏輯
- 測試邊界值與異常輸入

**Step 4 — 彙整報告**
整合三向審查結果，產出結構化報告。

## 輸出格式

```markdown
## 🔍 程式碼審查報告

### 需求摘要
[簡述使用者的需求]

### 審查結果

#### ✅ 符合需求的部分
- [正確實現的功能]

#### ❌ 問題清單
- **問題 1**：[描述]
  - 位置：[檔案, 行數]
  - 原因：[為什麼有問題]
  - 建議修改：[可直接複製貼上的修正]

#### ⚠️ 潛在風險
- [可能在特定條件下出錯的地方]

### 測試結果（如有執行）
| 測試案例 | 預期輸出 | 實際輸出 | 結果 |
|---------|---------|---------|------|

### 修正後程式碼
（僅在有問題時提供完整修正版本）
```

## 支援語言

Python（含 PyTorch / NumPy / Pandas）、C（含指標與記憶體管理）、MATLAB（含矩陣運算）

## 原則

- 若程式碼完全正確，直接說「符合需求，未發現問題」
- 不做超出需求範圍的「改進建議」
- 報告使用**繁體中文**
